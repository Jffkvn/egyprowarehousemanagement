Approval threshold rule (must be a real, working rule â€” not a placeholder)  **Default rule:** A request routes automatically to the **Warehouse Manager** for approval if all of the following are true:  - Requested quantity is fully available in stock, AND  - The item's `unit_value_ugx` is below **500,000 UGX**, AND  - Requested quantity leaves at least 1 unit remaining in stock after fulfillment (i.e., this is not the last unit)  Otherwise, the request routes to the **CFO**.  Store this threshold as a single editable setting (`approval_threshold_ugx`, default `500000`) in a `settings` table, not hardcoded in application logic, so the CFO can change it later from a Settings page without a code change.  ---  ## 6. Core workflows (build exactly this sequence)  ### 7.1 Receiving new equipment  Warehouse Manager (or CFO) fills a form: category, name, quantity (1 for reusable â€” repeat form per unit, or bulk quantity for consumable), unit value in UGX, manufacturer serial (optional), condition (Good / Pending Inspection â€” no "damaged" intake flow in Phase One, that's a Phase Two addition). On submit: creates the `equipment` or `consumable_stock` row(s), generates the asset/SKU code, and logs a `stock_added` transaction.  ### 7.2 Submitting a request (PM)  PM picks project name, site, needed-from date, needed-until date (only shown/required if any selected item is reusable), and adds one or more items from a searchable list showing current availability. On submit, the system checks the approval threshold rule (Section 5) against every line item and sets `routed_to` accordingly â€” if any single line item requires CFO approval, the whole request routes to CFO.  ### 7.3 Approving a request  The routed approver (WM or CFO) sees it in their "Pending Approvals" queue, with full request detail, and can Approve or Reject with a required reason if rejecting. Approval does not move stock yet â€” it only changes `requests.status` to `approved`. Physical handoff is a separate step (7.4), because approval and the equipment actually leaving the building are not always the same moment.  ### 7.4 Checkout (fulfilling an approved request)  Warehouse Manager opens an approved request and confirms the physical handoff: for each reusable item, this creates a `checkout` transaction and flips `equipment.status` to `checked_out`; for each consumable, this creates a `stock_consumed` transaction and decrements `quantity_on_hand`. Request status becomes `fulfilled`.  ### 7.5 Return (reusable items only)  Warehouse Manager logs the return: selects the checked-out item, records condition (Good / Damaged / Missing Parts / Non-Functional) and a notes field. This creates a `return` transaction and updates `equipment.status` to `available` (if good) or `under_repair` (if damaged/non-functional/missing parts). Request status becomes `returned` once all its reusable items are back. Damage report as a separate linked entity is a Phase Two addition â€” in Phase One, the condition + notes on the return transaction is sufficient and is not lost, just not yet its own table.  ### 7.6 Overdue detection  A scheduled check (daily) flags any `equipment` row where `status = 'checked_out'` and the linked request's `needed_until` date has passed, setting `equipment.status` to `overdue`. This shows on the Warehouse Manager's dashboard as an alert list. No CFO escalation automation in Phase One â€” that's a Phase Two addition per the earlier design discussion.  ---  ## 8.
